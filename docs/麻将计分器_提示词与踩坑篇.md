## 二、开发中踩过的 7 个坑

### 坑 1：设置页的数字输入框删不掉「1」

**现象：** 用户想把底分从 1 改成 5，先删掉 1，输入框立刻又蹦回 1，怎么也删不掉。

**原因：** 微信小程序的 `input` 是受控组件。原来的代码在 `bindinput` 里做了最小值校验——空字符串触发 `isNaN → val=0 → 小于1 → 强制为1 → setData`，小程序检测到绑定值变化，立刻把输入框内容重置为 "1"，用户永远清不掉。

**解决：**
```javascript
// 错误写法：bindinput 里直接校验
onInput(e) {
  let val = parseFloat(e.detail.value);
  if (isNaN(val) || val < 1) val = 1;  // ❌ 空字符串直接被覆盖
  this.setData({ settings: { baseScore: val } });
}

// 正确写法：空字符串直接 return，等 bindblur 时再校验
onInput(e) {
  if (e.detail.value === '') { return; }  // ✅ 允许中间态
  const val = parseFloat(e.detail.value);
  if (isNaN(val)) return;
  this.setData({ settings: { baseScore: val } });
},
onBlur(e) {
  let val = parseFloat(e.detail.value);
  if (isNaN(val) || val < 1) val = 1;   // ✅ 离开输入框再补
  this.setData({ settings: { baseScore: val } });
}
```

**教训：** 微信小程序 `input` 不能在 `bindinput` 里强制覆盖用户正在输入的中间态。

### 坑 2：手动调分弹窗的输入框文字被截断

**现象：** 输入框 placeholder「输入调整分数（可负数）」只显示一半，像是被裁剪了。

**原因：** 微信小程序的 `input` 组件**不会根据内容自动撑开高度**，不设置 `height` 时默认极小，文字被截断。

**解决：**
```css
/* ❌ 错误：只设置 padding，没有显式高度 */
.manual-input {
  padding: 16rpx 20rpx;
  font-size: 28rpx;
}

/* ✅ 正确：显式设置 height + line-height */
.manual-input {
  height: 88rpx;
  line-height: 88rpx;
  padding: 0 24rpx;
  font-size: 30rpx;
}
```

**教训：** 微信小程序 `input` 必须显式设 `height`，否则文字高概率显示不全。

---

### 坑 3：庄标和重置按钮重叠

**现象：** 玩家卡片右上角，「庄」圆圈和「重置」文字按钮叠在一起，点不准。

**原因：** 「重置」用了 `position: absolute; top: 14rpx; right: 10rpx;`，而「庄」标也在右上角，两者没有做空间分配。

**解决：** 将两者放入同一个 flex 行，去掉 absolute 定位：

```xml
<!-- 名称行：[名字✏] flex-1  [庄]  [重置] -->
<view class="player-name-row">
  <view class="player-name-tap">...</view>
  <view class="zhuang-badge">庄</view>      <!-- 中间 -->
  <view class="reset-player">重置</view>    <!-- 右侧 -->
</view>
```

**教训：** flex 布局比 absolute 定位更可靠，小卡片内慎用 absolute。

---

### 坑 4：底部操作按钮被导航栏遮挡

**现象：** 页面最下方的「撤销」「清空本局」按钮，有一半被固定底部导航栏覆盖，无法完整看到。

**原因：** 固定定位的底部导航（`position: fixed`）不占文档流高度，但页面 `padding-bottom` 只有 140rpx，不够撑开滚动空间。

**解决：**
```css
/* 底部 padding 必须大于导航栏实际高度 */
/* 导航栏约 86rpx + safe area，留 200rpx 最保险 */
.page {
  padding: 0 0 200rpx;  /* ✅ 从 140 改为 200 */
}
```

**教训：** 有固定底部导航时，页面 `padding-bottom` 要比导航实际高度多留 50-80rpx 的余量。

---

### 坑 5：历史记录字段名拼写不一致

**现象：** 历史页面「连庄」信息不显示，数据明明存了但就是读不到。

**原因：** `index.js` 保存历史记录时用的是 `lianZhuangCount`，但 `history.wxml` 读取时用的是 `item.lianZhuang`，字段名对不上。

```javascript
// index.js 写入
histRec.lianZhuangCount = lianZhuangCount  // ✅ 存的是这个

// history.wxml 读取
{{item.lianZhuang}}  // ❌ 拼错了字段名
```

**解决：** 统一改为 `item.lianZhuangCount`。

**教训：** 多个文件协同开发时，字段命名要保持一致，建议统一在 `calc.js` 里定义常量。

---

### 坑 6：0 值被 `||` 运算符误判

**现象：** 庄家 id 为 0（东家）时，重新进入小程序后庄家总是变成默认值，存储失效。

**原因：**
```javascript
// ❌ 错误写法：0 被 || 当成 falsy 值，等同于 null
this.globalData.zhuangJiaId = saved.zhuangJiaId || 0;
// 当 zhuangJiaId = 0 时，0 || 0 = 0（看似对，但若存储返回 '' 则出问题）
```

**解决：**
```javascript
// ✅ 正确写法：明确判断 null/undefined
this.globalData.zhuangJiaId = saved.zhuangJiaId != null ? saved.zhuangJiaId : 0;
```

**教训：** 凡是值可能为 0 的字段，绝对不能用 `||` 设默认值，要用 `!= null` 判断。

---

### 坑 7：UI 里出现技术术语让用户看不懂

**现象：** 手动调分弹窗里有一行警告：「⚠ 手动调分不进入撤销栈」，用户反馈看不懂「撤销栈」是什么意思。

**解决：** 直接把术语翻译成后果描述：

```
❌ 手动调分不进入撤销栈
✅ 此操作不可撤销，请确认分数无误
```

**教训：** 面向普通用户的 App，所有提示文案都要用「结果语言」而非「技术语言」。

## 三、总结

| 坑 | 类型 | 核心原因 |
|---|---|---|
| input 删不掉 1 | 微信组件特性 | 受控输入框不能在 bindinput 强制覆盖 |
| 输入框文字截断 | 微信组件特性 | input 必须显式设 height |
| 按钮重叠 | 布局问题 | absolute 定位缺乏空间规划 |
| 底部遮挡 | 布局问题 | padding-bottom 没预留足够空间 |
| 字段名不一致 | 协同问题 | 多文件开发字段命名不统一 |
| 0 值被误判 | JS 基础 | 用 `!=null` 而非 `||` 判断零值 |
| 技术术语 | 产品文案 | 用结果描述替代技术描述 |
